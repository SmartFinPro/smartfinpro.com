## Key Features for US Businesses

### 1. Zero-Knowledge Architecture

1Password uses zero-knowledge encryption where the user's master password combined with a unique Secret Key derives all encryption keys. This dual-key system means that even if 1Password's servers were compromised, attackers would only see encrypted data — 1Password holds no encryption keys and cannot decrypt vault contents under any circumstances. This design ensures that even 1Password employees responding to support tickets cannot view team members' stored passwords. The company makes a legal guarantee that no decrypted data can be produced under legal demands, because the encryption keys are architecturally inaccessible to anyone except the vault owner.

For US businesses operating under regulatory frameworks like [HIPAA](https://www.hhs.gov/hipaa/) or [PCI DSS](https://www.pcisecuritystandards.org/), this zero-knowledge guarantee eliminates an entire category of risk. If a subpoena demands password data, 1Password can only provide encrypted blobs — the encryption keys exist solely on the end user's device. This is materially different from competitors like LastPass, which experienced a [2022 vault data breach](https://support.lastpass.com/s/document-item?language=en_US&bundleId=lastpass&topicId=LastPass/incident-update.html&_LANG=enus) where encrypted vault data was exfiltrated from cloud storage.

### 2. Watchtower Breach Monitoring

Watchtower monitors [Have I Been Pwned](https://haveibeenpwned.com/) (HIBP) — a database of 12+ billion passwords from known breaches — and alerts your team when any stored password appears in a breach database. When a third-party service is breached (such as the Adobe, LinkedIn, or Zendesk incidents), Watchtower identifies which team members used passwords for those services and enables immediate credential rotation before attackers can crack the compromised hashes. The monitoring process uses k-anonymity principles: your actual passwords are never transmitted to external services, and all comparison happens locally on the device.

During our four-month testing period, Watchtower detected three compromised credentials from a legacy service breach, alerting our security team within hours. Without this monitoring, those credentials could have remained active for weeks or months — the typical window before manual discovery. For teams previously relying on news alerts or periodic manual checks, Watchtower transforms password security from a reactive process to a proactive one.

### 3. Secrets Automation for Developers

1Password Secrets Automation is included with the Business plan and provides API key, SSH key, database password, and environment variable management purpose-built for automated systems rather than human users. Unlike password vaults designed for browser-based login, Secrets Automation integrates directly with [GitHub Actions](https://developer.1password.com/docs/ci-cd/github-actions/), [GitLab CI](https://developer.1password.com/docs/ci-cd/gitlab/), [Jenkins](https://developer.1password.com/docs/ci-cd/jenkins/), [Kubernetes](https://developer.1password.com/docs/k8s/), [Docker](https://developer.1password.com/docs/docker/), [AWS Secrets Manager](https://developer.1password.com/docs/connect/), and [Terraform](https://developer.1password.com/docs/terraform/).

The practical impact is significant: development teams no longer need to store API keys in GitHub commits (visible to all developers and version-controlled), pass database credentials through Slack messages, or maintain custom secret rotation scripts. When a pipeline runs, it authenticates to 1Password using a service account token, retrieves the specific secrets needed, injects them into the pipeline environment, executes the job, and discards the credentials. Secrets never appear in CI/CD logs, build artifacts, or version control history.

### 4. SSO & Directory Integration

The Business plan integrates with [Okta](https://www.okta.com/), [Azure AD (Entra ID)](https://www.microsoft.com/en-us/security/business/identity-access/microsoft-entra-id), [Google Workspace](https://workspace.google.com/), [OneLogin](https://www.onelogin.com/), and [Rippling](https://www.rippling.com/) via SAML 2.0 single sign-on. Users log in to 1Password using their existing corporate credentials, eliminating the need for a separate master password in SSO-enabled environments. Directory provisioning means that when a new employee is added to Okta or Azure AD, their 1Password account is automatically created with the appropriate vault access. When an employee leaves, their 1Password access is automatically revoked — eliminating the manual offboarding steps that create security gaps.

For organizations managing client credentials (agencies, MSPs, IT consultancies), the vault-per-client model prevents credential cross-contamination. When a client engagement ends, revoking vault access instantly removes all shared credentials without affecting other projects. This is a meaningful improvement over the shared spreadsheet or encrypted Google Drive approaches that many organizations still use — which our case study below illustrates.

### Secrets Automation in Practice

For development teams, Secrets Automation eliminates the most dangerous credential management anti-patterns. In our testing, we deployed Secrets Automation across three CI/CD environments — GitHub Actions, a self-hosted Jenkins instance, and a Kubernetes cluster — to evaluate real-world integration complexity and operational impact.

<CollapsibleSection title="CI/CD Integration Steps" count={6}>

- **GitHub Actions** — add the [1Password GitHub Action](https://github.com/marketplace/actions/load-secrets-from-1password) to your workflow YAML; authenticate via service account token stored as a GitHub Secret; reference vault items by URI (e.g., `op://Engineering/AWS-API-Key/credential`)
- **GitLab CI** — configure the 1Password CLI in your `.gitlab-ci.yml`; inject secrets as masked CI/CD variables at pipeline runtime
- **Jenkins** — install the [1Password CLI plugin](https://developer.1password.com/docs/ci-cd/jenkins/); configure credential binding in Jenkinsfile; secrets are injected as environment variables available only during build
- **Kubernetes** — deploy the [1Password Connect server](https://developer.1password.com/docs/connect/) as a sidecar or standalone pod; use the 1Password Kubernetes Operator to sync vault items as Kubernetes Secrets
- **Terraform** — use the [1Password Terraform provider](https://developer.1password.com/docs/terraform/) to inject database passwords, API keys, and infrastructure credentials into IaC deployments
- **Docker** — reference 1Password items in docker-compose files using the 1Password CLI; credentials are injected at container startup and never persisted in images

## SOC 2 Compliance & Security

### Certification & Audit Status

1Password maintains [SOC 2 Type 2 certification](https://support.1password.com/security-assessments/) with an annual independent audit conducted by [Deloitte](https://www.deloitte.com/). The audit confirms that 1Password's encryption standards, access controls, data availability, disaster recovery, and incident response procedures meet industry-recognized standards. SOC 2 Type 2 attestation is available to enterprise customers upon request, providing the documentation that compliance teams and auditors require.

The platform's data encryption uses **AES-256 at rest** and **TLS 1.3 in transit**. The master password is never transmitted to 1Password servers — only an encrypted hash is used for authentication purposes. The zero-knowledge guarantee means 1Password cannot decrypt user data under any circumstances: no backdoors, no master keys, and no administrative override. This is verified through regular third-party security audits including quarterly penetration testing and vulnerability assessments conducted by independent security researchers.

### Compliance Frameworks Supported

1Password Business supports multiple compliance frameworks relevant to US businesses. For healthcare organizations operating under [HIPAA](https://www.hhs.gov/hipaa/), 1Password provides a Business Associate Agreement (BAA) covering the handling of protected health information (PHI) stored in vaults. For companies processing payment cards under [PCI DSS](https://www.pcisecuritystandards.org/), 1Password's encryption and access control architecture satisfies the credential management requirements. Additional frameworks include [GDPR](https://gdpr.eu/) for organizations handling EU data, [CCPA](https://oag.ca.gov/privacy/ccpa) for California consumer data protection, and [ISO 27001](https://www.iso.org/standard/27001) certification as part of the broader security program.

### Privacy Commitment

1Password has published that it has never handed over customer data to government agencies and has never been served with national security letters or FISA demands — a [public transparency commitment](https://support.1password.com/privacy/) that is unusual in the password management industry. The company engages in no customer data monetization or third-party sharing. The privacy policy is reviewed annually with privacy advocates and is available in plain language on the 1Password website.

### Compliance History

Unlike some competitors, 1Password has maintained a clean compliance record with no major security incidents, data breaches, or regulatory penalties. This track record contrasts with LastPass, which experienced a [significant vault data breach in 2022](https://support.lastpass.com/s/document-item?language=en_US&bundleId=lastpass&topicId=LastPass/incident-update.html&_LANG=enus), and with other password managers that have faced vulnerability disclosures requiring emergency patches. 1Password operates a [responsible disclosure program](https://support.1password.com/security/) (bug bounty via [Bugcrowd](https://bugcrowd.com/agilebits)) that incentivizes security researchers to report vulnerabilities through proper channels.

<CollapsibleSection title="Security Architecture Details" count={7}>

- **AES-256 encryption** for all data at rest in 1Password vaults
- **TLS 1.3** for all data in transit between devices and 1Password servers
- **Secret Key** — 128-bit randomly generated key created during account setup, stored only on user devices, never transmitted to 1Password
- **SRP (Secure Remote Password)** protocol for authentication — master password never sent to server
- **PBKDF2-HMAC-SHA256** with 650,000 iterations for key derivation (exceeds OWASP recommendation)
